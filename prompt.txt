## Problème

Dans `backend/app/workers/scan_tasks.py`, SQLMap tourne (`sqlmap_ran: true`) mais ne teste aucun paramètre (`sqlmap_params: []`, `sqlmap_vulnerable: false`) alors qu'il devrait au minimum tester les probe packs du pack `express_nodejs` (dont `/rest/user/login` avec le paramètre `email`).

## Diagnostic partiel déjà fait

Le pipeline a été refactorisé en fonctions imbriquées async (`_p3b_sqlmap`, `_p3a_*`, etc.) dans le cadre d'une parallélisation partielle. Deux problèmes ont été identifiés :

1. **`has_injectable_params` check supprimé dans `_p3b_sqlmap`** -- ce check bloquait SQLMap même quand les probe packs étaient disponibles. Il a été supprimé mais SQLMap retourne toujours `sqlmap_params: []`.

2. **Bloc fallback param probe** (autour de l'index 122000-124000) -- ce bloc synchrone utilise `loop.run_until_complete(_probe_param_endpoints())` pour sonder des endpoints avec paramètres sur la cible, puis injecte les résultats dans `ffuf_result["endpoints"]`. Ce bloc est conditionné par `if not has_injectable_params`. Or la ligne `has_injectable_params = True  # FIX G` est définie APRÈS ce bloc mais AVANT `def _p3b_sqlmap()` -- donc le fallback probe ne s'exécute jamais (le `if not has_injectable_params` est `False` au moment où il est évalué car `has_injectable_params` a été forcé à `True` avant).

3. **Probe packs** (`resolve_probes(probe_pack_ids, base_url)`) -- censés injecter des endpoints prioritaires (dont `/rest/user/login` avec paramètre `email`) dans la liste d'endpoints SQLMap. Ces probe packs sont normalement insérés dans `_call_sqlmap_enriched`, mais `sqlmap_params: []` suggère qu'ils ne sont pas pris en compte.

## Ce qu'il faut faire

### Diagnostic demandé en premier

Lis le code autour des sections suivantes dans `scan_tasks.py` et identifie précisément pourquoi `sqlmap_params` est vide :

1. La fonction `_call_sqlmap_enriched` -- vérifie qu'elle reçoit bien `probe_pack_ids` et qu'elle insère les probe packs dans `endpoints` avant de construire la requête SQLMap.
2. Le flux d'exécution depuis `_p3b_sqlmap()` → `_call_sqlmap_enriched(...)` -- vérifie que `ffuf_result`, `zap_result`, `katana_result` et `probe_pack_ids` ont des valeurs correctes au moment de l'appel (pas des dicts vides ou des exceptions).
3. La variable `has_injectable_params` et le bloc fallback probe -- vérifie l'ordre exact d'évaluation.

### Fix demandé

Une fois la cause identifiée, corrige pour que SQLMap reçoive au minimum les endpoints des probe packs (`express_nodejs` contient `/rest/user/login` avec paramètre `email`) -- ce qui suffit pour détecter le SQLi confirmé sur cette application.

La logique souhaitée est simple :
```
probe_pack_ids = agent_decision.get("probe_pack_ids") or ["generic_rest_api"]
probe_probes = resolve_probes(probe_pack_ids, base_url)
# Ces probes doivent être insérées EN PREMIER dans endpoints avant tout le reste
# pour que SQLMap les teste même si ZAP/FFUF n'ont trouvé aucun paramètre
```

Si le problème est que `_call_sqlmap_enriched` reçoit des résultats FFUF/ZAP vides (parce que les gather async ont retourné des exceptions ou des dicts vides), ajoute une validation défensive :
```python
# Si ffuf_result est vide/exception, utiliser un dict default
if not isinstance(ffuf_result, dict) or not ffuf_result.get("endpoints"):
    ffuf_result = {"endpoints": [], "by_severity": {}, "total": 0}
```
